## Description
Generate a comprehensive README.md inside .devcontainer/ that explains:

1. **Overview**: What this devcontainer does and why (Claude Code sandbox with network isolation)
2. **Prerequisites**: Docker, VS Code with Dev Containers extension
3. **Getting Started**: How to open the project in the devcontainer
4. **Firewall Architecture**: How iptables + ipset + dnsmasq work together, default DROP policy
5. **Adding Domains**: How to add new domains to the allowlist (static vs dynamic, edit dynamic-domains.conf)
6. **Claude Code Permissions**: Explains bypass mode and why it is safe inside the sandbox
7. **Settings Sync**: How sync-claude-settings.sh works and why it is needed
8. **Customization**: How to add port forwards, env vars, VS Code extensions, services
9. **Troubleshooting**: Common issues (firewall blocking needed domain, permission errors, volume issues)

Template should include the detected stacks and their specific domains for context.

## Implementation Plan

### Approach

Add a new `README.md.tmpl` template to `internal/render/templates/` and a corresponding `readme.go` render function that follows the established pattern (pure function, config in, bytes out). The README is a Markdown file rendered via `text/template` against `GenerationConfig`, making it dynamic based on detected stacks and their domain allowlists. The template includes all nine sections from the description, with stack-specific content rendered via `{{ range }}` over `Stacks`, `Domains.Static`, and `Domains.Dynamic`.

The approach mirrors how `Dockerfile.tmpl` and `devcontainer.json.tmpl` are structured: a `readme.go` file with a package-level `template.Must(template.ParseFS(...))` and a public `README(cfg GenerationConfig) (string, error)` function.

### Files to Create/Modify

- `internal/render/templates/README.md.tmpl` **(new)** -- The Markdown template with all nine sections. Uses `{{ range .Stacks }}` for detected-stacks list, `{{ range .Domains.Static }}` and `{{ range .Domains.Dynamic }}` for domain tables, and static prose for architecture, troubleshooting, etc.
- `internal/render/readme.go` **(new)** -- Render function `README(cfg GenerationConfig) (string, error)` that parses and executes `README.md.tmpl`. Follows the `dockerfile.go` pattern: package-level `template.Must(template.ParseFS(templateFS, ...))`, `bytes.Buffer`, return string.
- `internal/render/readme_test.go` **(new)** -- Structural assertion tests following the project's testing strategy (no golden files).

### Steps

1. **Create `internal/render/templates/README.md.tmpl`** -- Write the Markdown template with the following sections:

   - **Header**: `# .devcontainer` with a one-line summary mentioning ccbox-generated devcontainer for Claude Code.
   - **Overview**: Static prose explaining sandboxed Claude Code with network isolation. Includes a dynamic "Detected Stacks" bullet list via `{{ range .Stacks }}` showing which stacks were configured.
   - **Prerequisites**: Static list (Docker Desktop or Docker Engine, VS Code + Dev Containers extension, or compatible editor). No dynamic content.
   - **Getting Started**: Static instructions for opening in VS Code (Command Palette > "Dev Containers: Reopen in Container"). No dynamic content.
   - **Firewall Architecture**: Static prose explaining the three-layer approach (iptables default DROP + ipset hash:ip allowlist + dnsmasq for dynamic domains). Include a brief description of `init-firewall.sh` as the entry point. Two dynamic subsections:
     - **Static Domains** table rendered via `{{ range .Domains.Static }}` with columns: Domain, Rationale.
     - **Dynamic Domains** table rendered via `{{ range .Domains.Dynamic }}` with columns: Domain, Rationale. Explain dnsmasq re-resolution.
   - **Adding Domains**: Static instructions explaining how to edit `dynamic-domains.conf` for new dynamic domains and how to re-run the firewall script. Mention that static domains require regenerating via `ccbox init`.
   - **Claude Code Permissions**: Static prose about bypass mode being safe inside the sandbox due to network isolation.
   - **Settings Sync**: Brief static mention of `sync-claude-settings.sh` and its purpose (syncing Claude Code settings from host to container). Keep it short since `ccbox-v1zh` hasn't landed yet.
   - **Customization**: Static prose about editing `devcontainer.json` for port forwards, env vars, VS Code extensions, additional services (Docker Compose).
   - **Troubleshooting**: Static Q&A-style section covering: domain blocked by firewall (check `dynamic-domains.conf`, re-run `warmup-dns.sh`), permission errors (`sudo` available), volume mount issues, DNS resolution failures.
   - **Footer**: "Generated by ccbox -- do not edit manually." line, consistent with other generated files.

   Template design notes:
   - Use `{{ if .Stacks }}` guards around the detected-stacks list so empty configs produce valid Markdown without blank sections.
   - Use `{{ if .Domains.Static }}` and `{{ if .Domains.Dynamic }}` guards around domain tables.
   - Use `stripWildcard` funcMap (already defined in `firewall.go`) for domain display in tables. Since `README.md.tmpl` will be parsed by a different template set than the firewall templates, the funcMap needs to be registered on the readme template too. The simplest approach: define a local `readmeFuncMap` in `readme.go` with the same `stripWildcard` helper, or (better) extract the shared `funcMap` to a package-level variable in a shared location. Given that `funcMap` in `firewall.go` is already package-level and unexported, the cleanest path is to reference it from `readme.go` since both are in `package render`. The template parse call becomes `template.New("").Funcs(funcMap).ParseFS(...)`.
   - Domain names in the table should show the stripped form (no `*.` prefix) for readability.

2. **Create `internal/render/readme.go`** -- Implement the render function:

   ```
   package render

   var readmeTmpl = template.Must(
       template.New("").Funcs(funcMap).ParseFS(templateFS, "templates/README.md.tmpl"),
   )

   func README(cfg GenerationConfig) (string, error) {
       var buf bytes.Buffer
       if err := readmeTmpl.ExecuteTemplate(&buf, "README.md.tmpl", cfg); err != nil {
           return "", fmt.Errorf("render: execute README.md template: %w", err)
       }
       return buf.String(), nil
   }
   ```

   Notes:
   - Uses `templateFS` from `embed.go` (the shared `//go:embed templates/*` FS), not the narrower `templatesFS` from `firewall.go` (which only embeds firewall-specific templates).
   - Uses `funcMap` from `firewall.go` since `stripWildcard` is needed for domain display. Both files are in `package render` so this is a direct reference with no import cycle.
   - Returns `string` like `Dockerfile()` rather than writing to `io.Writer` like `DevContainer()`. The README is a small file; string return is simpler for the caller and matches the Dockerfile pattern. Either signature works, but consistency with the more common pattern in this codebase is preferred.

3. **Create `internal/render/readme_test.go`** -- Write structural assertion tests:

   - **`TestREADME_NoError`** -- Integration test: `Merge([Go], nil)` then `README(cfg)`, verify no error, verify non-empty output.
   - **`TestREADME_ContainsAllSections`** -- Verify output contains all nine section headings (e.g., "## Overview", "## Prerequisites", "## Getting Started", "## Firewall Architecture", etc.).
   - **`TestREADME_DetectedStacksListed`** -- Integration test with `Merge([Go, Node], nil)`: verify output contains "go" and "node" (or their display names).
   - **`TestREADME_StaticDomainsInTable`** -- Integration test: for each `cfg.Domains.Static`, verify `d.Name` appears in output. Spot-check `github.com`.
   - **`TestREADME_DynamicDomainsInTable`** -- Integration test: for each `cfg.Domains.Dynamic`, verify stripped domain name appears in output.
   - **`TestREADME_EmptyConfig`** -- Hand-built `GenerationConfig` with all empty (non-nil) slices. Verify: no error, no `<no value>` artifacts, still contains all static section headings, no empty Markdown tables (domain sections should be skipped or show a "none" message).
   - **`TestREADME_AllStacks`** -- Integration test with all five stacks: verify all stack names appear, every domain appears.
   - **`TestREADME_NoTemplateArtifacts`** -- Verify output does not contain `<no value>`, `{{`, `}}`, or `<nil>`.
   - **`TestREADME_GeneratedByFooter`** -- Verify output contains "Generated by ccbox".
   - **`TestREADME_Deterministic`** -- Render twice with same config, verify `bytes.Equal`.
   - **`TestREADME_WildcardDomainsStripped`** -- Verify `*.anthropic.com` appears as `anthropic.com` in the domain table (not the raw wildcard form).

### Testing Strategy

- **Two-tier testing** as established by the codebase rules:
  - Integration tests: go through `Merge` then `README`, validating the full pipeline.
  - Isolation tests: hand-built `GenerationConfig` with non-nil empty slices, custom stacks, or edge-case domain lists.
- **Structural assertions**: Check for section headings, domain presence, stack mentions -- never golden-file snapshots.
- **Registry-computed completeness**: Iterate `cfg.Domains.Static` and `cfg.Domains.Dynamic` and assert every entry appears in output (same pattern as `TestRenderFirewall_InitFirewall_ContainsStaticDomains`).
- **Spot-checks**: Assert well-known entries (`github.com`, `registry.npmjs.org`).
- **Empty-input safety**: Empty config produces valid Markdown without template artifacts.
- **No `<no value>` artifacts**: Explicit assertion on every test that renders output.
