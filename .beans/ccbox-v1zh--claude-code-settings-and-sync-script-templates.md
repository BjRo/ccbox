## Description
Two templates:

**claude-user-settings.json:**
```json
{
  "permissions": {
    "defaultMode": "bypassPermissions",
    "allow": ["Bash", "Read", "Write", "Edit", "Grep", "Glob", "Task", "WebFetch", "WebSearch"]
  },
  "enabledPlugins": [<stack-specific LSP plugins>]
}
```
The `enabledPlugins` array is parameterized: include `gopls-lsp` for Go, `typescript-lsp` for Node/TS, etc.

**sync-claude-settings.sh:**
Port the credfolio2 version as-is. This script:
1. Checks for template at `/workspace/.devcontainer/claude-user-settings.json`
2. If no existing settings: copies template
3. If settings exist: deep-merges with `jq -s '.[0] * .[1]'` (template wins)
4. Preserves runtime state written by Claude Code

This script is stack-agnostic — no parameterization needed, emit as static file.

## Implementation Plan

### Approach

Add two new template files and a single Go render file that exposes a `RenderClaude` function returning a `ClaudeFiles` struct. This follows the pattern established by `firewall.go` / `RenderFirewall`, grouping related templates behind one render function with a typed return struct.

The `claude-user-settings.json.tmpl` template is parameterized by `GenerationConfig.LSPs` (specifically the `Plugin` field). The `sync-claude-settings.sh.tmpl` template is static (no Go template actions) but still rendered through `text/template` for pipeline uniformity, consistent with how `warmup-dns.sh.tmpl` is handled.

**Resolved discrepancy:** The bean description mentions `gopls-lsp`, `typescript-lsp` style identifiers, but the actual `stack.LSP.Plugin` field values in the registry are `"gopls"`, `"typescript"`, `"pyright"`, `"rust-analyzer"`, `"solargraph"`. The template will use whatever values `GenerationConfig.LSPs` provides (sourced from the registry), not the examples in the bean description.

### Files to Create

- `internal/render/templates/claude-user-settings.json.tmpl` -- JSON template parameterized by `GenerationConfig.LSPs[].Plugin`
- `internal/render/templates/sync-claude-settings.sh.tmpl` -- Static shell script template (no Go template actions)
- `internal/render/claude.go` -- Render function and template parsing
- `internal/render/claude_test.go` -- Tests for both templates

### Files to Modify

None. The new render file is self-contained. The shared `embed.go` already embeds `templates/*` via a glob, so new `.tmpl` files are automatically included. No changes to `render.go` (Merge) or `GenerationConfig` are needed because `LSPs` is already in the struct.

### Steps

1. **Create `claude-user-settings.json.tmpl`** -- A JSON template that renders the Claude Code user settings file.
   - Static `permissions` block with `defaultMode: "bypassPermissions"` and the fixed `allow` array.
   - Dynamic `enabledPlugins` array built by ranging over `.LSPs` and emitting each `.Plugin` value.
   - JSON formatting: the template must produce valid JSON for any number of LSP entries (0, 1, or many). Use `{{range $i, $lsp := .LSPs}}` with a comma separator conditional on `$i > 0` to avoid trailing commas (JSON does not permit them).
   - When `.LSPs` is empty, `enabledPlugins` should be an empty array `[]`.

   Template structure:
   ```
   {
     "permissions": {
       "defaultMode": "bypassPermissions",
       "allow": ["Bash", "Read", "Write", "Edit", "Grep", "Glob", "Task", "WebFetch", "WebSearch"]
     },
     "enabledPlugins": [{{range $i, $lsp := .LSPs}}{{if $i}}, {{end}}"{{$lsp.Plugin}}"{{end}}]
   }
   ```

2. **Create `sync-claude-settings.sh.tmpl`** -- A static shell script that syncs Claude user settings at container startup.
   - Shebang: `#!/usr/bin/env bash` with `set -euo pipefail`
   - Header comment: purpose description + "Generated by ccbox -- do not edit manually."
   - Variables:
     - `TEMPLATE="/workspace/.devcontainer/claude-user-settings.json"` (the generated settings file)
     - `SETTINGS_DIR` pointing to `$HOME/.claude` (the Claude Code settings directory)
     - `SETTINGS_FILE` pointing to `$SETTINGS_DIR/settings.json` (the Claude Code runtime settings file)
   - Logic:
     - Check that `$TEMPLATE` exists; if not, warn and exit 0 (non-fatal).
     - Create `$SETTINGS_DIR` if it does not exist (`mkdir -p`).
     - If `$SETTINGS_FILE` does not exist: copy template directly (`cp`).
     - If `$SETTINGS_FILE` exists: deep-merge using `jq -s '.[0] * .[1]'` where `.[0]` is the existing settings and `.[1]` is the template. Template wins on conflicts. Write to a temp file, then `mv` atomically.
   - No Go template actions. The script is stack-agnostic.

3. **Create `claude.go`** -- Render function for the two Claude-related templates.
   - Parse templates at package level using the shared `templateFS` from `embed.go`:
     ```go
     var claudeTemplates = template.Must(
         template.ParseFS(templateFS, "templates/claude-user-settings.json.tmpl", "templates/sync-claude-settings.sh.tmpl"),
     )
     ```
     This uses the shared `templateFS` (not a new `//go:embed` directive), consistent with the approach in `devcontainer.go`. Since there are no custom FuncMap helpers needed (no `stripWildcard` or similar), plain `ParseFS` suffices.
   - Define `ClaudeFiles` struct:
     ```go
     type ClaudeFiles struct {
         UserSettings     []byte // claude-user-settings.json content
         SyncSettings     []byte // sync-claude-settings.sh content
     }
     ```
   - Implement `RenderClaude(cfg GenerationConfig) (ClaudeFiles, error)`:
     - Execute `claude-user-settings.json.tmpl` into a `bytes.Buffer` with `cfg` as data.
     - Execute `sync-claude-settings.sh.tmpl` into a second `bytes.Buffer` with `cfg` as data.
     - Return `ClaudeFiles` with both buffers' bytes.
     - Wrap errors with `fmt.Errorf("render claude-user-settings.json: %w", err)` / `fmt.Errorf("render sync-claude-settings.sh: %w", err)`.

4. **Create `claude_test.go`** -- Tests following the structural assertion pattern.

   **Integration tests (through Merge + RenderClaude):**
   - `TestRenderClaude_NoError` -- Merge Go+Node, render, assert both fields are non-empty.
   - `TestRenderClaude_UserSettings_ValidJSON` -- Merge Go+Node, render, `json.Unmarshal` the UserSettings output. Must produce valid JSON.
   - `TestRenderClaude_UserSettings_Permissions` -- Parse JSON, verify `permissions.defaultMode` is `"bypassPermissions"`, verify `permissions.allow` contains all 9 tools.
   - `TestRenderClaude_UserSettings_PluginsMatchRegistry` -- Merge all 5 stacks. For each `cfg.LSPs`, assert its `.Plugin` value appears in the parsed `enabledPlugins` array. This is a structural assertion computed from the registry, not a hardcoded list.
   - `TestRenderClaude_UserSettings_PluginsSpotCheck` -- Merge Go+Node. Assert `enabledPlugins` contains `"gopls"` and `"typescript"` (spot-checks for well-known values).
   - `TestRenderClaude_UserSettings_EmptyStacks` -- Merge with empty stacks. Assert `enabledPlugins` is `[]` (empty array), not `null` or absent. JSON must still be valid.
   - `TestRenderClaude_UserSettings_NoDuplicatePlugins` -- Merge Go+Go (duplicate). Assert each plugin appears exactly once in `enabledPlugins`.
   - `TestRenderClaude_UserSettings_Deterministic` -- Render twice with same config, assert output is byte-equal.

   **Sync script tests:**
   - `TestRenderClaude_SyncSettings_ScriptStructure` -- Assert shebang `#!/usr/bin/env bash`, `set -euo pipefail`, reference to `claude-user-settings.json`, reference to `.claude/settings.json`, `jq` merge command.
   - `TestRenderClaude_SyncSettings_IsStatic` -- Render with Go-only config and Go+Node+Python config. Assert `SyncSettings` output is byte-equal (script is stack-agnostic).
   - `TestRenderClaude_SyncSettings_NoTemplateArtifacts` -- Assert no `<no value>` strings in output.

   **Isolation tests (hand-built GenerationConfig):**
   - `TestRenderClaude_DirectConfig_EmptyLSPs` -- `GenerationConfig` with `LSPs: []stack.LSP{}`. Assert valid JSON with `"enabledPlugins": []`.
   - `TestRenderClaude_DirectConfig_CustomPlugins` -- `GenerationConfig` with synthetic LSP entries. Assert custom plugin names appear in output.

### Testing Strategy

- **Two-tier** per project convention: integration tests go through `Merge` then `RenderClaude`; isolation tests use hand-built `GenerationConfig`.
- **Structural assertions** computed from registry data for completeness (iterate `cfg.LSPs` and assert each `.Plugin` appears).
- **Spot-checks** for well-known values (`"gopls"`, `"typescript"`).
- **JSON validity** via `json.Unmarshal` for the settings template.
- **Static script verification** by rendering with different configs and asserting byte-equality.
- **No golden files**.

### Architecture Decision

No new ADR needed. This follows the existing embedded template rendering pattern (ADR-0006) without any new architectural decisions. No new FuncMap helpers, no new dependencies, no new patterns.

### Open Questions

None. The approach is straightforward and all decisions are grounded in existing codebase patterns.
