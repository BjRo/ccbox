## Context & Motivation

Users want both Claude Code and OpenAI Codex available in the same devcontainer — one as lead coder, one as reviewer. Today, agentbox is Claude-only. Adding Codex requires touching installation, settings, firewall, VS Code extensions, volume mounts, and the stack plugin registry.

### Codex CLI Key Facts (from research)

- **Package**: `@openai/codex` (npm) — also available as standalone binary
- **Config location**: `~/.codex/config.toml` (TOML format)
- **Auth**: ChatGPT interactive login (primary) OR `OPENAI_API_KEY` env var
- **VS Code extension ID**: `openai.chatgpt`
- **Approval modes**: `--ask-for-approval` flag, `--full-auto` convenience, `--dangerously-bypass-approvals-and-sandbox` (alias `--yolo`)
- **Sandbox modes**: `read-only` (default), `workspace-write`, `danger-full-access`
- **API domain**: `api.openai.com`
- **No LSP plugin system**: Codex does not have Claude-style LSP plugins

## Implementation Plan

### Phase 1: Refactor Plugin Field to Be Tool-Agnostic

**Problem**: `stack.LSP.Plugin string` is Claude-specific. With Codex (and potentially future tools), we need a generic structure.

**Change `internal/stack/stack.go`**:
- Replace `Plugin string` in `LSP` struct with `Plugins map[string]string` where keys are coding tool identifiers (e.g., `"claude"`, `"codex"`) and values are the plugin identifiers.
- Define constants for tool keys: `CodingToolClaude = "claude"`, `CodingToolCodex = "codex"`.
- Update all stack registry entries:
  - Go: `Plugins: map[string]string{"claude": "gopls-lsp@claude-plugins-official"}`
  - Node: `Plugins: map[string]string{"claude": "typescript-lsp@claude-plugins-official"}`
  - Python: `Plugins: map[string]string{"claude": "pyright-lsp@claude-plugins-official"}`
  - Rust: `Plugins: map[string]string{"claude": "rust-analyzer-lsp@claude-plugins-official"}`
  - Ruby: `Plugins: map[string]string{}` (empty map, no plugins for either tool)
- Update `copyStack()` to deep-copy the map via `maps.Clone`.

**Update `internal/render/render.go`**:
- `GenerationConfig.LSPs` still contains `[]stack.LSP` — the `Plugins` map carries through.

**Update `internal/render/claude.go`** and **`claude-user-settings.json.tmpl`**:
- Template must iterate `LSPs` and extract only the `"claude"` key from each `Plugins` map.
- Add a `claudePlugin` FuncMap helper: `func(plugins map[string]string) string { return plugins["claude"] }`.
- Template becomes: `{{range $i, $lsp := .LSPs}}{{$p := $lsp.Plugins | claudePlugin}}{{if $p}}...{{end}}{{end}}`

**Update all tests** referencing `LSP.Plugin`.

### Phase 3: Codex Settings Template & Sync Script

**Create `internal/render/templates/codex-config.toml.tmpl`**:
- Static TOML config for full-auto mode in an externally sandboxed container:
  ```toml
  # Generated by agentbox -- do not edit manually.
  # Codex CLI configuration for sandboxed devcontainer.
  approval_policy = "unless-allow-listed"
  sandbox_mode = "danger-full-access"
  ```
- Note: `approval_policy = "unless-allow-listed"` + `sandbox_mode = "danger-full-access"` is the Codex equivalent of Claude's `bypassPermissions` — the container itself provides the sandbox.

**Create `internal/render/templates/sync-codex-settings.sh.tmpl`**:
- Static shell script, mirrors `sync-claude-settings.sh.tmpl` pattern.
- Syncs `codex-config.toml` → `~/.codex/config.toml`.
- Uses TOML-aware merge or simple copy (TOML merge is harder than JSON — may just overwrite on first run and skip if exists, since Codex config is simpler).

**Create `internal/render/codex.go`**:
- `CodexFiles` struct with `Config []byte` and `SyncSettings []byte`.
- `RenderCodex(cfg GenerationConfig) (CodexFiles, error)` — similar to `RenderClaude`.
- Parse both templates at package level with `template.Must`.

**Update `cmd/init.go` `renderFiles()`**:
- Add `codex-config.toml` and `sync-codex-settings.sh` to the file manifest.

**Update `cmd/update.go`**:
- Handle new Codex files in update flow.

**Tests**:

## Open Questions

1. **Codex approval_policy value**: Need to verify exact TOML enum value for "no approvals needed." Candidates: `"never"`, `"on-request"`, `"unless-allow-listed"`. The `--full-auto` flag maps to `"on-request"` + `workspace-write` sandbox. For a fully sandboxed container, `--dangerously-bypass-approvals-and-sandbox` is more appropriate, but its config.toml equivalent needs verification.
2. **Auth domains**: Need to verify whether `auth.openai.com` or `auth0.openai.com` is used for ChatGPT login flow, or if it's handled through `api.openai.com`.
3. **Codex telemetry domains**: Claude has `sentry.io` and `statsig.com`. Codex may have its own telemetry endpoints that need firewall allowlisting.
