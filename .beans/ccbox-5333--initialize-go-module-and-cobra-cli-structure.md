## Description
Initialize the Go module (`github.com/bjro/ccbox`), set up Cobra CLI with a root command and `init` subcommand. Establish the project directory structure:

```
cmd/
  root.go        # Root Cobra command
  init.go        # `ccbox init` subcommand
internal/
  detect/        # Stack detection
  render/        # Template engine (renamed from template/ to avoid stdlib shadow)
  firewall/      # Domain allowlist logic
  config/        # .ccbox.yml handling
main.go
```

Use Go 1.24+ with modules. Add a basic `--version` flag.

## Implementation Plan

### Approach

Bootstrap the Go module and Cobra CLI skeleton with minimal but correct structure. This bean is purely scaffolding: create the module, wire up Cobra with a root command and an `init` subcommand stub, add version reporting, and create placeholder packages under `internal/` so downstream beans have a clear place to land. No business logic -- just the bones.

We use `cobra` (no `cobra-cli` code generator) and wire commands manually. This keeps dependencies minimal and the code fully under our control. The version will be injected via `ldflags` at build time (GoReleaser-friendly) with a hardcoded fallback of `dev`.

Key decisions informed by Challenge Report feedback:
- **Renamed `internal/template/` to `internal/render/`** to avoid shadowing `text/template` and `html/template` from stdlib. The package name `render` describes the action (rendering templates into files) and reads naturally: `render.DevContainer(...)`.
- **Unexported `newRootCmd()` constructor** for test isolation. Tests call it per-test to get a fresh command tree, eliminating shared state issues. The package-level `rootCmd` calls it once at init. This is one function and establishes the correct testing pattern from day one.
- **Internal test package `cmd`** (not external `cmd_test`) since tests need access to the unexported `newRootCmd()` constructor. True black-box CLI testing is better served by integration tests in a later bean.
- **Dropped duplicate `TestInitCommand_AppearsInHelp`** since the root help test already covers subcommand visibility.

   - Package `cmd`

### Testing Strategy

- **Unit tests in `cmd/` package**: Use Cobra's test support (`SetArgs`, `SetOut`) via the `newRootCmd()` constructor for per-test isolation. Internal test package (`package cmd`) gives access to the unexported constructor.
- **Fresh command per test**: Every test calls `newRootCmd()` to get a clean command tree. No shared state, no cleanup needed, no fragile reset logic.
- **No tests for placeholder packages**: They contain no logic. Tests will be added by downstream beans when functionality lands.
- **Build verification**: `go build ./...` and `golangci-lint run ./...` as part of the implementation step, not as separate test files.

### Design Decisions

- **Manual Cobra wiring (no cobra-cli generator)**: Keeps code simple and avoids generated boilerplate. We control the exact structure.
- **`version` as package variable with ldflags injection**: Standard Go pattern. GoReleaser sets this automatically. Default of `"dev"` means `go install` from source works without special flags.
- **Unexported `newRootCmd()` constructor**: Builds a fresh command tree per call. The package-level `rootCmd` calls it once at init for production use. Tests call it per-test for isolation. This is the standard Cobra testing pattern and establishes the correct approach for all downstream beans.
- **`newInitCmd()` called from `newRootCmd()`, no `init()` function**: Command registration is explicit in the constructor, not scattered across `init()` functions. This makes the command tree construction traceable and testable.
- **Internal test package (`package cmd`)**: Tests need access to the unexported `newRootCmd()` constructor. This is the right tradeoff for a CLI package where the public API is just `Execute()`.
- **`internal/render/` instead of `internal/template/`**: Avoids shadowing `text/template` and `html/template` from stdlib. The package will heavily use `text/template` internally, so avoiding the name collision eliminates import aliasing friction across the entire codebase.
- **Placeholder packages with doc comments only**: Establishes directory structure without dead code. Each package is a valid Go package that compiles, so `go build ./...` works from day one.

### Findings

#### Go Engineer

> **Finding 1: Double error output from `main.go` and Cobra** (severity: WARNING)
>
> Step 3 says `main.go` should "print to stderr and exit with code 1" on error from `Execute()`. However, Cobra's `Execute()` already prints errors to stderr by default before returning them. If `main.go` also prints the error, every command failure will produce duplicate error messages. For example, an unknown subcommand would print:
>
> ```
> Error: unknown command "foo" for "ccbox"
> Error: unknown command "foo" for "ccbox"
> ```
>
> **Option A (recommended):** Set `SilenceErrors: true` on the root command in `newRootCmd()` and let `main.go` handle all error printing. This gives `main.go` full control over error formatting.
>
> **Option B:** Remove the error printing from `main.go` entirely -- just call `os.Exit(1)` on non-nil error. Let Cobra handle all error output. This is the minimal-code approach.

> **Finding 2: Missing `SilenceUsage: true` on root command** (severity: WARNING)
>
> Step 4 does not set `SilenceUsage: true` on the root command. By default, Cobra prints the full usage/help text on every error. This means a simple typo like `ccbox inti` will dump the entire help output above the error message. This is the single most common Cobra setup mistake and creates a poor user experience from day one.
>
> **Option A (recommended):** Add `SilenceUsage: true` to the root command in `newRootCmd()`. Users who want help can explicitly pass `--help`. Error messages stay clean.
>
> **Option B:** Set `SilenceUsage` dynamically in a `PersistentPreRun` hook so it only silences after argument parsing. This is heavier than needed for a scaffolding bean.

> **Finding 3: `init()` in `cmd/root.go` contradicts the explicit-construction principle** (severity: SUGGESTION)
>
> Step 4 uses `init()` to set the package-level `rootCmd`, but step 5 explicitly avoids `init()` for `initCmd`, stating: "no `init()` function, no package-level variable" and "this keeps command registration explicit and the construction flow clear." The same reasoning applies to `rootCmd` itself. A package-level `var rootCmd = newRootCmd()` achieves the same result without `init()`, and the construction order is visible at the declaration site rather than hidden in an `init()` function.
>
> **Suggestion:** Replace `init()` with `var rootCmd = newRootCmd()` at package level. This aligns `root.go` with the explicit-construction philosophy already established for `init.go`.
