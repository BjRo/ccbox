# .devcontainer

This devcontainer was generated by [agentbox](https://github.com/bjro/agentbox) to run Claude Code in a sandboxed environment with network isolation.

## Overview

This setup creates a Docker-based development container with:

- Full Claude Code installation with bypass permissions (safe due to network isolation)
- iptables-based firewall with default DROP policy
- DNS-aware domain allowlisting via dnsmasq
- Language tooling installed via [mise](https://mise.jdx.dev/)

## Firewall Architecture

The container uses a three-layer network isolation approach:

1. **iptables default DROP** -- All outbound traffic is blocked by default
2. **ipset hash:net allowlist** -- Only resolved IPs of explicitly allowed domains can be reached
3. **dnsmasq** -- Continuously re-resolves dynamic domains and updates the ipset

The `init-firewall.sh` script sets up all three layers at container startup. Static domains are resolved once and their IPs are added to the allowlist. Dynamic domains are managed by dnsmasq, which updates the allowlist as DNS records change.

### Static Domains

These domains have stable IPs and are resolved once at container startup:

| Domain | Rationale |
|--------|-----------|
| sentry.io | Error reporting for Claude Code |
| statsig.com | Feature flags and experimentation for Claude Code |

### Dynamic Domains

These domains have rotating IPs (CDNs, load balancers) and are continuously re-resolved by dnsmasq:

| Domain | Rationale |
|--------|-----------|
| anthropic.com | Anthropic API - required for Claude Code to function |
| api.github.com | GitHub REST API - required for git clone/push/pull over HTTPS |
| api.openai.com | OpenAI API - required for Codex CLI to function |
| auth.openai.com | OpenAI auth - required for Codex ChatGPT login flow |
| github.com | GitHub web and git-over-HTTPS |
| proxy.golang.org | Default Go module proxy - serves go get / go mod download |
| storage.googleapis.com | GCS backend for Go module proxy - actual module content delivery |

## Adding Domains

2. Add a new line: `domain.example.com  # Rationale for allowing this domain`
3. Rebuild the container or run `sudo /usr/local/bin/warmup-dns.sh` inside the container

**Static domains** require regenerating the devcontainer setup via `agentbox init`.

## Claude Code Permissions

Claude Code runs in **bypass mode** inside this container, meaning it has full permissions to read/write files, execute commands, and access the network. This is safe because:

- The container's firewall restricts all network access to explicitly allowed domains
- The container filesystem is isolated from your host (only the workspace is mounted)
- No sensitive credentials are available inside the container by default

## Customization

### Dockerfile Custom Stage

The Dockerfile uses a multi-stage build:

1. **`agentbox` stage** -- Managed by agentbox. Contains system packages, runtimes, LSPs, Claude Code, and firewall scripts. Regenerated by `agentbox update`.
2. **`custom` stage** -- Managed by you. Add project-specific tools here. Preserved by `agentbox update`.

Add your customizations in the `custom` stage at the bottom of the Dockerfile:

```dockerfile
FROM agentbox AS custom
RUN go install github.com/user/tool@latest
RUN npm install -g some-cli
```

## Troubleshooting

**A domain is being blocked by the firewall**

- Check if the domain is listed in `.devcontainer/dynamic-domains.conf`
- Add it if missing, then run `sudo /usr/local/bin/warmup-dns.sh` to update the allowlist
- For persistent changes, re-run `agentbox init` to regenerate with the domain included

**Permission errors inside the container**

- The default user has passwordless `sudo` access
- Prefix commands with `sudo` when elevated permissions are needed

**Volume mount issues**

- Ensure Docker has permission to access the project directory
- On macOS, check Docker Desktop file sharing settings
- On Linux, verify the project directory permissions

**DNS resolution failures**

- Run `dig +short domain.example.com` to test resolution
- Check that dnsmasq is running: `sudo systemctl status dnsmasq`
- Review firewall logs: `sudo iptables -L -v -n`
